# MrSample — v0.2 followups

Captured during v0.1 shipping (2026-05-16).

## Loop point granularity — RESOLVED for v0.1 (2026-05-16)

Original step was 0.01 (1% = ~440 frames at 44.1 kHz). Fixed in v0.1
post-build:

- `sample_start`, `loop_start`, `loop_end` step → `0.001` (~44 frames per click)
- `shift_increment_multiplier` → `0.1` so Shift+jog-click → `0.0001` (~4 frames)
- The dedicated knob row already runs through `knob_engine.mjs` which
  applies divisor 1..16 acceleration based on tick gap, giving slow-slow
  rotations sample-accurate resolution (step/16 = 0.0000625).

If users still want frames-based UI, revisit; for now this is enough.

## Auto-refresh dependent params after file-picker commit

When a WAV with a `smpl` chunk loads, the plugin auto-seeds `loop_mode`,
`loop_start`, `loop_end`, and `root_note` inside `set_sample_path`. The
chain UI doesn't re-poll these after a filepath browser commit — value
cache invalidates only on slot/view/level transitions
(`shadow_ui.js:8634`-ish). So the menu displays stale values until you
back out and re-enter the Sample level.

Workarounds considered:
- `browser_hooks.on_commit` actions exist but only set static values
  (no "$loop_start" template that reads back from plugin).
- Add a host-side hint like `"refresh_after_commit": ["loop_mode",
  "loop_start", "loop_end", "root_note"]` in `chain_params` that
  invalidates those cache entries after commit. Host-side change.

Document as v0.1 caveat ("re-enter Sample to see auto-seeded loop
points"). Implement host-side `refresh_after_commit` in v0.2.

## Other deferrals

- Async sample loading (worker thread). Currently load is synchronous on
  the audio thread's set_param caller, which is fine for short samples
  but will stutter on large FLAC/MP3 decodes. MrDrums has the same trade.
- Pitch LFO (currently LFO routes to filter only).
- Filter envelope as a separate ADSR (currently filter env = amp env × amount).
- Multi-zone / round-robin (out of scope for "simple sampler"; that's SFZ territory).
